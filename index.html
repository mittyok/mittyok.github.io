REST-API-DESIGN-RULEBOOK

2017-11-12

API设计最佳实践

避免在URI中使用下划线_

  • 在有的应用字体中，下划线并不是很明显，甚至有时候会被完全隐藏掉，看不到。典型的，比如浏览器会把链接用下划线标识以给使用者可点击的暗示，这个时候下划线就会完全被盖掉
  • 中划线本意有连字符的含义，google会把通过中划线连接的词看为两个词对待，而把下划线连接的词看为一个词对待

尽量使用小写字母在URI的PATH中

RFC 3986中有这样的定义，URI中，scheme及host以外的部分均是大小写敏感的，所以如果我们又有大写又有小写会给设计带来不必要的困扰

在URI中尽量不要包含文件后缀名

我们不能通过在URI中包含文件后缀的方式表明消息体的格式，而是应该通过media type来指明

为说有的API使用一致的子域名

如：api.baidu.com

为所有的开发之portal提供一致的子域名

如：developer.baidu.com

资源原型

document

类似一个对象实体或者是数据库记录

collection

一组document，服务端控制管理

store

客户端控制管理的资源仓库，服务端不需要创建新的URI，客户端决定资源的URI，通过自己的URI存和取

controller

具有程序化概念的URI，比如邮件的再次发送：mails/12345/resend

URI中Path部分的设计

{collection-c}/{store-s}/{document-d}

单数名词应该用作document的名字

复数名词应该用作collection的名字

复数名词应该用作store的名字

动词或者动词短语应该用作controller的名字

URI Path的变量部分应该用标识ID来替换

如：users/{userId}/roles/{roleId}

CRUD动作名称不要出现在URI的Path中

POST：新增
PUT：修改
DELETE：删除
GET：查询

比如，对于删除，不建议使用的URI：
GET /deleteUser?id=1234
GET /deleteUser/1234
DELETE /deleteUser/1234
POST /users/1234/delete

URI查询部分的设计

RFC 3986 中定义，可选查询部分应该位于Path之后，在fragment之前

分页查询相关参数应该再查询部分体现

GET和POST不要与其它methods混用

GET必须用于获取资源内容

HEAD应该在只需要获取响应头时使用

PUT必须用于insert或者update store的资源

PUT必须用于更新可变的资源

POST必须用于创建新的资源

POST必须用于执行controller

DELETE必须用于移除资源

OPTIONS应该用于获取资源可用的methods

Selenium Grid

2017-07-18

Selenium-Grid是什么

Selenium-Grid可以让我们在不同机器的不同浏览器上并行的运行测试。也就是说，我们可以同时在不同的机器、不同的操作系统，不同的浏览器上同时运行测试。本质上，Selenium-Grid支持分布式的测试执行。它允许我们在一个分布式的执行环境中运行测试。

什么时候用Selenium-Grid

一般来说，有两个原因我们会用Selenium-Grid：

  1. 我们需要在不同的操作系统，多种类型或者版本的浏览器中运行我们的测试
  2. 为了减少我们执行测试套件所花费的时间

启动Selenium-Grid

一般来说，我们需要先启动一个hub，应为所有的nodes是依赖hub的。

Start a Hub

java -jar selenium-server-standalone-2.44.0.jar -role hub

Start a Node

java -jar selenium-server-standalone-2.44.0.jar -role node  -hub http://localhost:4444/grid/register

Demo(with Ruby)

可以看下这个demo，比较清晰。

Trouble shooting

404 page not found error

一种比较隐蔽的情形：

比如：我们通过域名访问hub，http://seleniumhub.com，selenium remote connect在初始化时，会对域名进行解析，用解析后的IP进行后续的连接测试操作。

好啦，问题来了：
假设我们的hub使用了反向代理：seleniumhub.com –> x.x.x.x:4444

我们的域名服务器将seleniumhub.com解析为：y.y.y.y（反向代理服务器在这台机子上）
selenium会使用解析后的ip y.y.y.y去连接hub进行测试，明显不可以y.y.y.y

解决办法：

from selenium.webdriver.common.desired_capabilities import DesiredCapabilities
from selenium.webdriver.remote.remote_connection import RemoteConnection

executor = RemoteConnection("http://seleniumhub.com/wd/hub", resolve_ip=False)
chrome = webdriver.Remote(executor, desired_capabilities=DesiredCapabilities.CHROME)

Fluentd vs Logstash

2017-07-18

说起Logstash，大家可能略显生疏，但是说起ELK，大家应该再熟悉不过。所谓的ELK即Elasticsearch，Logstash，Kibana。在Panda Strike公司，他们使用了ELK工具，他们有自己的Elasticsearch集群，但是逐渐的，他们去掉了Logstash，而是使用了Fluentd替换了Logstash，称其为EFK，虽然听起来挺拗口的，但是我们还是看一下他们为什么要这么做。

Why

首先呢，Fluentd和Logstash均提供了日志处理（Log shipper）和转发（Log forwarder）的功能。Wikipedia中对log shipping有定义：

log shipping “is the process of automating the backup of a database and transaction log files on a primary (production) database server, and then restoring them onto a standby server.”

对log forwarder也有定义：

log forwarder “is a forestry vehicle that carries big felled logs from the stump to a roadside landing.”
单单根据定义，我们无法看出两者的差别，因此，在严格的线上产品中，我们宁可什么也不用，也不能根据维基的定义来引入框架。

尽管如此，Panda Strike选择了另外的定义：

Log shipper在现代的devops中是一个必要的组件，因为日志是流，而不是文件。
Log forwarder是吧日志事件发送给Log shipper。作为forwarder，它的任务就是把上游的日志事件尽可能快的发送出去，而shipper会决定如何处理和路由这些事件。Shipper有可能会对事件做一些聚合操作，或者直接发送给远端的存储或者分析工具。

Fluentd和Logstash有一些共同的特性，而且对于部分特性的支持非常的类似：

  • 都可以给不同的层提供必要的功能
  • 在任何时候，forwarders都能够检测和发现失败的shipper，进而切换到另一个可用的shipper
  • 都可以转发日志事件到不同的接收者，比如说：SQS、Elasticsearch、S3等
  • 都原生支持JSON格式的log事件
  • 目前都在有人维护和开发，并且都有企业资助
  • 都使用的插件架构，并且都有自己的插件管理工具
  • 都支持日志事件的过滤，转换，同时插件的数量上也趋于相近

Fluentd主要是用Ruby写的，有一些性能敏感的部分是用C写的，同时有一个方便的、稳定的预编译版本可用。Fluentd还用一个用Go写的forwarder。Logstash的fowwarder也是用Go写的，但是它的shipper是运行在JRuby上的，这就需要JVM的支持。这种依赖我们觉得对log shipping来说是一种比较重的依赖。

在其他方面也有一些差别：

  • 可靠性、持久性、故障恢复
  • 可调优性，数据的flush间隔，工作线程的一些配置等
  • 内存和CPU的一些需求（对我们而言，就这一点Fluentd就轻松的胜出了）

再来一个删节版的输出看一下：

# LOGSTASH FORWARDER
$ ps fauxwww --sort -rss |awk '/(USER|logstash-forwarder)/ { print }'
%CPU %MEM
0.0  0.7

# LOGSTASH SHIPPER
ps fauxwww --sort -rss |awk '/(USER|logstash)/ { print }'
%CPU %MEM
4.8 20.9

# FLUENTD FORWARDER
ps fauxwww --sort -rss |awk '$1 ~ /(USER|td-agent)/ { print }'
%CPU %MEM
0.0  2.3
0.1  4.5

# FLUENTD SHIPPER
ps fauxwww --sort -rss |awk '$1 ~ /(USER|td-agent)/ { print }'
%CPU %MEM
0.0  2.3
0.2  7.3

这些数据完全来自我们的生产环境，不是基于某些方面的一些基准测试。从数据上看，Logstash的forwarder明显胜出了Fluentd的Forwarder。但是Fluentd的shipper的表现要远远好于Logstash的shipper，并且在对Logstash令人失望的经验之后，Fluentd使用起来更容易，更可靠。
